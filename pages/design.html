Detailed Component Design

This section explains the internal design of Hidden In Plain Sight. Since the application runs entirely on the Android device, the design focuses on local data handling, core application functions, and the steganography algorithm used to embed and extract hidden messages.

Data Storage Design

Hidden In Plain Sight does not use a database or remote server. This is an intentional design decision that supports offline field use and reduces the risk of sensitive message retention.

Data Item Stored? Description
JPEG Image Yes The selected or captured image is stored locally on the device and is used as the base image for embedding or extraction.
Secret Message No The plaintext message is entered by the user for embedding, but it is discarded after processing is complete.
Stego Image Yes The newly generated steganographic image is saved locally so it can be shared through normal communication channels.
Extracted Message No The recovered hidden message is displayed to the user but is not written to storage after extraction.

Core Application Functions

The following functions represent the core logic of the Hidden In Plain Sight system. These functions handle authentication, image selection, and the embedding and extraction of hidden messages.

inputGesture()

Input: User gesture

Output: Success | ErrorCode

Validates the user's hidden unlock gesture against the stored credentials. If successful, the application transitions from the disguised interface to the hidden messaging system.

selectImage()

Input: Photo library

Output: PlainImage | ErrorCode

Allows the user to select a JPEG image from local device storage using the Android photo picker.

embedMessage()

Input: Message, PlainImage

Output: StegoImage | ErrorCode

Sends the image and message to the native C++ layer through the JNI bridge, where the steganography algorithm modifies JPEG DCT coefficients to encode the hidden message.

extractMessage()

Input: StegoImage

Output: Message | ErrorCode

Sends the steganographic image to the native C++ layer so the algorithm can read the modified DCT coefficients and reconstruct the hidden message.

Algorithm Logic

Our algorithm is designed to embed messages before JPEG compression, which differs from many traditional approaches. A JPEG image is broken into small 8x8 pixel blocks and transformed into frequency data using the discrete cosine transform (DCT). These values are then quantized during compression. Instead of waiting until after compression, our system modifies selected DCT coefficients before compression is finalized.

Embedding Algorithm

embedMessage(image, message):

  convert message into a binary bit sequence

  load JPEG image

  select a set of mid-frequency DCT coefficients

  avoid very low frequencies
  avoid very high frequencies

  slightly adjust selected coefficients
  to encode the message bits

  allow normal JPEG compression to occur

  rebuild and save the steganographic image

  discard the original plaintext message

Mid-frequency coefficients are used because very low frequencies can create visible image distortion, while very high frequencies are more likely to be destroyed during JPEG compression. This makes mid-frequency values the best location for hiding the message while preserving image quality.

Extraction Algorithm

extractMessage(image):

  load steganographic JPEG image

  read the same selected DCT coefficients

  reconstruct the hidden binary message

  convert binary data back into text

  display the recovered message to the user

  do not store the extracted message

The extraction process reverses the embedding process by reading the same coefficient locations that were modified earlier. The hidden bit sequence is reconstructed and converted back into a readable message.
